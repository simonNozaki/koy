# ADR-0001: プレイグラウンドのアーキテクチャ

## ステータス

Accepted

## コンテキスト

Koy 言語に Array・Set・Object リテラルなどの機能拡張を行う前に、ブラウザ上でコードを手軽に試せるプレイグラウンド環境を整備する必要がある。

プレイグラウンドに求められる要件は以下の通り。

- ブラウザからコードを入力して実行できる
- 標準出力の結果をブラウザ上に表示できる
- エラー発生時にはエラー内容をブラウザ上に表示できる
- 既存の Koy インタープリタ（Kotlin）を再利用できる

REPL（Read-Eval-Print Loop）方式は、インタープリタの状態管理・入力解析・エラーリカバリが複雑になるため採用しない。

## 決定

以下の構成でプレイグラウンドを実装する。

### モジュール構成の位置づけ

`koy-playground` は Gradle サブプロジェクトとして `project(":")` 経由でコアに依存する。
`Interpreter` および `Parsers` を直接利用するが、これは外部への不正な結合ではなく、**playground がインタープリターの実行環境を提供するサーバーサイドコンポーネントである**という整理に基づく。
コアの公開 API（`Interpreter` / `Parsers`）を使うことは意図された依存関係であり、JAR をサブプロセスとして起動する方式は採用しない（後述）。

### バックエンド

- **Ktor embedded server**（Kotlin）を使用する
- `POST /run` エンドポイントを実装し、リクエストボディのコード文字列を受け取ってインタープリタで評価する
- 標準出力のキャプチャには `Interpreter` のコンストラクタに `PrintStream` を注入する方式を採用する
  - `System.setOut` はグローバル状態を変更するためスレッドセーフでなく、複数リクエストが並行して処理される Web サーバ環境では使用できない
  - `ByteArrayOutputStream` をラップした `PrintStream` をコンストラクタ経由で渡すことで、リクエストごとに独立した出力ストリームを確保する
- エラー発生時はスタックトレース文字列をレスポンスの `error` フィールドに格納してフロントエンドへ返す

### フロントエンド

- **Vite + Vue 3** でビルドする
  - 当初はフレームワークなしの Vite + TypeScript で構成したが、コンポーネントが増えるにつれて `main.ts` 1ファイルへの責務集中が読みづらくなったため、Vue 3 SFC に移行した
- コードエディタには **CodeMirror 6** を使用する（シンタックスハイライト対応）
- ビルド成果物は Ktor の `resources/static/` に配置し、Ktor が静的ファイルとして配信する

### シンタックスハイライト

Koy 言語のシンタックスハイライトには CodeMirror 6 の **`StreamLanguage`** を採用する。

`StreamLanguage` はストリームを1文字ずつ読み進めながらトークン種別を返すシンプルな API であり、独自言語への対応に適している。CodeMirror 6 が提供するもう一つのアプローチとして **Lezer パーサー**（`@lezer/generator`）があるが、不採用とした。

| 方式 | 概要 | 採用可否 |
|---|---|---|
| `StreamLanguage` | 正規表現ベースのトークナイザーを定義する。実装が軽量で、文法が単純な言語に向く | ✅ 採用 |
| Lezer パーサー | 専用の文法ファイル（`.grammar`）を書き、ビルドツールでパーサーを生成する。AST を持つため補完・エラー表示などへの拡張性が高い | 現時点では過剰 |

Koy の構文は制御フロー・関数定義・リテラル程度にとどまるため、現時点では `StreamLanguage` で十分な表現力がある。将来的に LSP 連携や補完機能が必要になった場合は Lezer パーサーへの移行を検討する。

### API インターフェース

```
POST /run
Content-Type: application/json

{ "code": "<Koy ソースコード>" }

---

200 OK
{ "output": "<標準出力文字列 or null>", "error": "<エラーメッセージ or null>" }
```

## 採用しなかった案

| 案 | 不採用の理由 |
|---|---|
| Fat JAR + Node/Python ラッパー | 言語をまたいだ構成になる。プロセス起動コストが発生する。 |
| CLI + ファイル監視 | ブラウザから直接操作できない。 |
| `System.setOut` によるキャプチャ | グローバル状態を書き換えるためスレッドセーフでなく、並行リクエストで出力が混在するリスクがある。 |
| ThreadLocal ベースのカスタム PrintStream | 実装が複雑になる。コンストラクタ注入で十分に目的を達成できる。 |
| REPL 方式 | インタープリタの状態管理・入力解析・エラーリカバリが複雑になる。 |
| フレームワークなし（Vite + TypeScript のみ） | 当初採用したが、エディタ・出力・状態管理が `main.ts` に集中して可読性が低下したため Vue 3 SFC に移行した。 |
| Lezer パーサーによるシンタックスハイライト | 文法ファイルのビルドが必要で実装コストが高い。Koy の構文規模では `StreamLanguage` で十分。 |

## 結果

- Koy インタープリタの変更は `Interpreter` のコンストラクタに `PrintStream` を受け取る引数を追加するだけに留まり、既存コードへの影響を最小化できる。
- フロントエンドは Vite + Vue 3（SFC）で構成し、エディタ・出力・状態管理をコンポーネント単位で分離して保守性を高める。
- Ktor が静的ファイル配信とAPIサーバを兼ねるため、デプロイ成果物が単一の JAR に収まる。
- スレッドセーフな設計により、将来的なリクエスト並行処理にも対応できる。

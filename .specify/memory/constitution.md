# Koy 言語 コンスティテューション

## コア原則

### I. 仕様ファースト
新しい言語機能はコードを書く前に仕様書を作成する。
仕様は構文・意味論・エッジケースを定義する。実装は仕様に従う。

### II. 式指向（非交渉）
すべての制御フロー構文（`if`、`while`、`for-in`）は式であり、ブロックの最後に評価された値を返す。
文のみの構文は導入しない。ブロックは常に値を生成する。

### III. 最小かつ一貫した構文
言語の構文は最小限に保つ。新しい構文は以下を満たす必要がある:
- 既存の構文と一貫性がある（例: 関数リテラルは `|args| { body }`）
- `Parsers.kt` の PEG 文法で曖昧さを生じない
- 識別子と衝突するキーワードを導入しない

### IV. デフォルト不変
`val` で宣言した変数はイミュータブル。再代入可能な変数には明示的に `mutable val` が必要。
新機能は `mutable` なしに変更を許してはならない。

### V. テストファースト実装
すべての言語機能は、マージ前に `InterpreterTests.kt` とパーサーテストを用意する。
テストは仕様のアクセプタンスシナリオに対応させる。

## 実装上の制約

- **パーサー**: `Parsers.kt` に parsecj を用いた PEG 文法で記述。新構文はここに追加する。
- **AST**: すべての式ノードは `Expression.kt` の sealed class として定義。新構文には新しい data class が必要。
- **インタープリタ**: 評価ロジックは `Interpreter.kt`。新しい AST ノードはここで処理する。
- **値**: ランタイム値の型は `Value.kt` で定義。

## 開発ワークフロー

1. `specs/<機能名>.md` を仕様テンプレートに従って作成
2. 仕様のアクセプタンスシナリオに対応する失敗テストを作成
3. `Expression.kt` → `Parsers.kt` → `Interpreter.kt` の順に実装
4. 全テストがパスしたら PR を作成
